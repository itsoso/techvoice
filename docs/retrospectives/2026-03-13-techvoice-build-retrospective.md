# TechVoice 系统构建完整复盘

日期：2026-03-13  
项目：TechVoice 匿名反馈系统  
范围：从 PRD 输入、前后端 MVP 实现、双仓库托管、线上部署、持续修复、样式迭代、数据清理，到过程复盘与最佳实践沉淀

## 1. 项目目标

本次工作的目标不是补一个小功能，而是把一份 PRD 变成一个可运行、可部署、可管理、可公开展示的匿名反馈系统。

最终要解决的问题包括：

- 员工可以匿名提交吐槽与提案
- 员工可以通过追踪码持续查看状态并继续补充
- 管理员可以登录、查看、回复、改状态、公开内容
- 公开回音壁可以展示被公开的反馈并接受点赞
- 系统可以独立部署，Nginx 代理，HTTPS 可用
- 代码需要同时进入 GitHub 和公司内网 GitLab

## 2. 最终交付结果

最终交付的是一个前后端分离、Python 技术栈的 MVP：

- 前端：React + Vite + TypeScript
- 后端：FastAPI + SQLAlchemy
- 数据库：SQLite
- 代理与部署：Nginx + systemd + Let's Encrypt
- 站点地址：[https://techvoice.executor.life/](https://techvoice.executor.life/)
- 代码仓库：
  - GitHub：`git@github.com:itsoso/techvoice.git`
  - GitLab：`git@git.corp.kuaishou.com:plateco-dev/kwaishop-product/kwaishop-techvoice-service.git`

当前系统支持：

- 匿名吐槽
- 结构化提案
- 追踪码查询
- 匿名补充回复
- 管理员登录
- 管理员回复
- 状态流转
- 发布到回音壁
- 回音壁展示公开内容、管理员回复、匿名补充、点赞
- 公开架构页
- 深浅主题切换

## 3. 完整构建过程

### 3.1 需求输入与范围收敛

项目起点是 PRD 文档，随后补充了一个 HTML 原型作为员工端视觉参考。  
一开始最关键的不是写代码，而是先把范围收敛清楚，否则后面会在“到底做 MVP 还是做全量 PRD”上不断返工。

最终收敛出的范围是：

- 做 `MVP`
- 管理员端不接 SSO
- 外部系统集成先不做真接入，只做接口预留或 mock
- 优先把员工端、管理员端、回音壁三条主链路跑通

### 3.2 技术选型

最终选型是：

- 后端：FastAPI
- ORM：SQLAlchemy
- 数据库：SQLite
- 前端：React + Vite
- 鉴权：JWT

这个组合的优势是开发速度快、边界清晰、MVP 成本低，适合这次“从 0 到 1”的交付目标。

### 3.3 MVP 实现

第一阶段完成的是基础链路：

- 员工匿名提交反馈
- 员工拿到 `thread_code`
- 员工追踪查询

- 管理员发布到回音壁

这一阶段构成了最小闭环，也为后续所有问题修复提供了主干。

## 5. 我们踩过的坑

### 5.1 范围没有先钉死，后面会变成“边做边问边返工”

一开始如果不先确认：

- MVP 还是全量版
- 是否接 SSO
- 哪些外部集成是真做，哪些只 mock

后面实现很容易反复重做。

### 5.2 Python 版本与服务器环境不一致

本地/项目元数据最初要求过高版本，服务器实际是 `Python 3.12.3`。  
这直接造成了部署阻塞。

教训：

- 部署前先看服务器运行时
- 项目声明版本不能只按本地环境写

### 5.3 前端把 API 地址写死成 `127.0.0.1`

这会导致线上页面访问时实际请求用户自己机器的 `127.0.0.1`，最终出现：

- `Failed to fetch`
- 登录失败
- 所有接口像“服务挂了”

根因是环境隔离没有处理好。

### 5.4 请求 header 合并错误导致后台 POST 变成 422

这是典型的前端请求封装 bug：

- `Content-Type: application/json` 被覆盖
- 后端拿不到合法 JSON body
- Admin 回复、改状态、发布全部失败

这个问题后来也被 code review 明确指出。

### 5.5 前端修了，线上却没有立即生效

这个坑反复出现了多次。原因不是代码没改，而是：

- 构建出了新 bundle
- 但线上入口 HTML 没切到新 bundle
- 或者浏览器缓存了旧资源

教训：

- 每次部署必须核对服务器上的 `index.html`
- 不能只看本地 `dist`
- 不能只说“已部署”，要验证线上资源引用

### 5.6 主题默认值被旧缓存拖偏

虽然代码后来改成默认深色，但旧浏览器 localStorage 里留了 `system`，用户看到的仍然可能是浅色或跟随系统。  
这说明“默认值修改”不等于“旧用户自动迁移”。

### 5.7 弹层被下层内容遮住

这类问题本质上是：

- 层叠上下文
- `z-index`
- `backdrop-filter`
- 父容器的 stacking context

如果只调弹层自身，不处理容器层级，问题会一直反复。

### 5.8 UI 在“微调字号”里来回震荡

这次顶部导航和各页标题做了很多轮微调，说明前期没有一次性定好：

- 品牌层级
- 主导航层级
- 工具按钮层级
- 页面标题层级

只靠“再大一点”“再小一点”会导致体验越来越碎。

### 5.9 回音壁一开始没有展示完整上下文

缺失内容包括：

- 原始完整内容
- 管理员回复
- 员工后续匿名补充

根因是公开接口只返回了部分序列化字段，而不是产品语义上的完整沟通链路。

### 5.10 生产数据操作时工具假设不成立

清理数据库时踩到的坑：

- 服务器没有 `sqlite3` 命令
- shell quoting 导致脚本执行失败
- 删除前如果不备份，风险太高

这属于典型的“本地经验不能直接套服务器”。

### 5.11 本地测试环境并不总是完整

后端某次测试没跑通，不是代码错，而是本地环境缺少 `bcrypt`。  
这说明“测试失败”不一定是业务失败，也可能是运行环境不完整。

## 7. 我从中学到的东西

### 7.1 设计决策越晚冻结，后面视觉迭代越容易失控

这次导航和标题字号来回调了很多轮。  
根因不是 CSS 难写，而是没有一开始就把下面这些定义清楚：

- 品牌是否应该高调
- 主导航和工具入口是否同级
- 页面标题和卡片标题的层级关系

### 7.2 “部署成功”必须建立在真实校验上

以后凡是说“已部署”“已修复”，必须至少验证：

- 线上 `index.html` 是否引用新 bundle
- 线上后端服务是否实际重启
- 线上接口是否返回期望数据

不能只看本地构建成功。

### 7.3 对外公开展示的数据模型不能只按内部接口思路设计

内部详情页和公开墙是两种产品语义：

- 内部详情页关注完整原始数据和操作
- 公开墙关注“可公开传播的上下文完整度”

如果公开接口只返回一部分字段，公开墙一定会缺信息。

### 7.4 旧数据和旧缓存迁移必须视为正式功能

改默认主题、改请求方式、改公开结构，如果不考虑：

- 历史 localStorage
- 历史数据库数据
- 历史 bundle 缓存

就会出现“代码对了，用户还错着”的情况。

### 7.5 线上运维也需要工程习惯

包括：

- 先备份再删数据
- 明确列出影响的表
- 做完后再核对计数
- 不依赖服务器一定有本地常用工具

## 8. 给你的建议

### 8.1 以后做类似项目，先把这五件事一次性定死

- 范围：MVP 还是全量
- 鉴权：SSO 还是本地账号
- 外部依赖：真接入还是 mock
- 部署：单机还是容器
- 设计风格：稳重产品感还是更强技术感

### 8.2 让“验收标准”比“需求文本”更具体

比如不要只写“支持回音壁”，要写成：

- 显示原始内容
- 显示管理员回复
- 显示员工后续补充
- 支持点赞
- 不暴露敏感标识

### 8.3 把“上线验证清单”固定成脚本或标准流程

建议以后固定包含：

- 本地测试
- 本地构建
- 远端部署
- 服务健康检查
- 线上 HTML 入口校验
- 线上接口抽样校验

### 8.4 视觉类需求最好不要靠“再大一点/再小一点”反复迭代

更高效的方式是一次性明确：

- 品牌字号
- 导航字号
- 页面主标题字号
- 卡片标题字号
- 正文字号

然后按这个层级表做，不要每次局部调。

## 9. 仍需关注的残留风险

这次做了很多修复，但仍有一些代码级风险值得后续继续收口：

- JWT secret 默认值在代码里仍有兜底值，虽然线上 env 已覆盖，但代码层仍建议强制要求环境变量
- `thread_code` / `public_code` 的生成和碰撞重试策略仍可继续增强
- 点赞计数目前仍适合继续做原子更新优化
- 后端 schema 的长度约束和空白内容校验仍可继续加强
- 一些 code review 提到的匿名隔离细节仍建议做正式安全审计

## 10. 最佳实践清单

下面是这次复盘沉淀出来、以后可以直接复用的最佳实践。

### 10.1 需求与范围

- 先确认 `MVP / Full`，不要边做边定义
- 先确认 `真实集成 / mock`
- 先确认 `登录方式 / 权限模型`
- 先确认 `上线目标环境`

### 10.2 技术选型

- 从 0 到 1 的内部工具优先选“开发快、边界清晰”的组合
- 前后端分离时，环境变量和 API base 必须从第一天就处理正确
- 生产环境 Python/Node 版本要先探测，再定项目约束

### 10.3 前端工程

- API client 统一封装，但必须有请求头单测
- 主题切换要同时考虑“默认值”和“旧缓存迁移”
- 导航、主题、弹层、面包屑属于框架级组件，要尽量少改结构、多做一次成型
- 标题层级需要一张统一的 typography map

### 10.4 后端工程

- 公开接口和内部接口不要共用完全一样的输出模型
- 时间线类系统要从产品视角定义“哪些事件应该公开可见”
- 输入校验不要只放前端，后端必须兜底
- 状态流转要有明确约束，不要靠页面逻辑暗含

### 10.5 部署与运维

- 每次部署必须验证线上 `index.html`
- 每次重启服务后必须看服务状态
- 每次接口修复后至少做一次线上抽样验证
- 删除线上数据必须先备份
- 线上不要假设有 `sqlite3`、`rg`、`jq` 等工具

### 10.6 Git 与交付

- 明确双仓库提交流程，不要临时想
- 提交前先跑测试和构建
- 公司仓库如果有作者邮箱要求，要提前固化
- 分支策略和提交策略要一致，不要一会推 `main` 一会推 feature

### 10.7 过程协作

- 需求确认型 prompt 和修 bug 型 prompt 要分层处理
- 对“看起来没生效”的问题，要优先怀疑部署链路和缓存，而不是先怀疑实现
- 长链路任务要持续沉淀复盘，不然同样的坑会重复踩

## 11. 以后如果再做一次，我会怎么做

如果重新从头做一次，我会把流程压缩成下面顺序：

1. 用 PRD + HTML 原型先定 MVP 边界
2. 一次性定技术栈、部署环境、登录方式
3. 先做员工端、管理员端、回音壁的最小闭环
4. 第一次上线前就准备好部署校验清单
5. 视觉层级先定稿，再做页面细节
6. 公开接口和内部接口一开始就区分
7. 所有后续迭代先判断是“代码问题、部署问题、缓存问题、数据问题”中的哪一类

这样后面的返工会少很多。
